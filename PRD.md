## 0. 本版结论

BiteBack 首发不做“饭点附近推荐”，不做“进店推券”，不做“美食地图”，也不做一个独立 App。首发只做一个窄切口：

> 用户在抖音搜索美食后，15 分钟内回到推荐流，但没有完成 POI、团购或路线行为时，系统用 AI 从用户历史收藏中召回高置信、强匹配、当前可行动的美食 POI，生成一张自然出现在 Feed 中的收藏唤醒卡。

这个切口的核心价值是：**把抖音已经沉淀的内容种草资产，转成搜索后未完成决策场景下的本地生活决策资产。**

首发只验证一件事：

> 对“搜索后未完成决策”的用户，BiteBack 能否在不伤害 Feed 主体验的前提下，提升 POI 决策行为和 7 日增量核销。

## 1. 产品定义

### 1.1 一句话

BiteBack 是抖音推荐流中的 AI 收藏美食唤醒卡，在用户搜索美食但没有完成决策后，把其过去收藏过且与当前搜索意图强匹配的美食内容，重新组织成一张可理解、可行动、可验证增量的本地生活卡片。

### 1.2 不是

- 不是附近随机推荐。
- 不是饭点通用推荐。
- 不是商家广告位。
- 不是电子围栏 / WiFi / 蓝牙到店推券。
- 不是独立美食地图或本地生活 App。
- 不是用低价券强行拉转化的商业卡。

### 1.3 是

- 是搜索后未完成决策场景的 Feed 承接实验。
- 是用户历史收藏兴趣与当前搜索意图的 AI 匹配。
- 是抖音信息流内的轻决策卡片。
- 是内容种草到本地生活履约之间的前置决策层。

## 2. 背景问题

抖音已经具备美食内容、达人探店、POI、团购券、搜索、榜单、核销等基础能力。真实断点不是“没有供给”，而是用户链路中间断掉：

```text
刷到美食视频
  -> 收藏 / 点赞 / 评论 / 搜索
  -> 后续真的要吃饭
  -> 在抖音搜索但没有完成决策
  -> 用户可能跳到美团 / 大众点评 / 高德重新决策
  -> 抖音失去本地生活转化机会
```

BiteBack 要补的是搜索后的决策缺口，而不是重新做一个美食推荐系统。

## 3. P0 首发人群

### 3.1 P0 定义

P0 只服务：

> 搜索后 15 分钟内回到推荐流，query 具备美食决策意图，搜索链路内没有完成 POI、团购或路线行为，且存在 S/A/B 级收藏 POI 与 query 在商圈、品类、店名至少一维强匹配的用户。

### 3.2 为什么选这个人群

| 判断 | 说明 |
| --- | --- |
| 意图最明确 | 搜索行为比饭点、定位、外出停留更能证明用户正在做决策 |
| Feed 风险最低 | 用户刚主动表达需求，卡片是承接，不是突然打断 |
| 归因更清晰 | 可对比搜索后回流展示 BiteBack 与不展示的转化差异 |
| Demo 更成立 | 信息流里刷到时，用户能立刻理解“我刚搜过，所以出现它” |

### 3.3 P1 / P2 暂不首发

| 场景 | 处理 |
| --- | --- |
| 饭点外出收藏唤醒 | P1，等 P0 验证后扩展 |
| 高收藏低转化人群主动唤醒 | P1，需更强 Feed 护栏 |
| 深夜美食收藏回响 | P2，更偏情绪体验，转化归因弱 |
| 纯附近推荐 | 不做 |
| 进店推券 | 不做，交给现有到店营销能力 |

## 4. Eligible UV 口径

Eligible UV 是实验分母，必须严格定义，避免用样本选择制造虚假 lift。

```text
Eligible UV =
搜索后 15 分钟内回到推荐流
AND query 命中美食/餐厅/品类/商圈决策意图
AND 搜索链路内未完成 POI 打开、团购页打开、支付、路线点击
AND 用户过去 180 天内有可绑定 POI 的美食收藏
AND 收藏 POI 与 query 在商圈/品类/店名至少一维强匹配
AND 收藏等级为 S/A/B，非 C 级云收藏
AND poi_confidence >= 0.85
AND 店铺当前营业或即将营业
AND 距离/商圈可达
AND 用户有位置相关授权
AND 当日未曝光 BiteBack
AND 未命中广告疲劳、隐私负反馈、低质商家风控
```

## 5. 用户需求与核心假设

### 5.1 JTBD

当我在抖音里搜索晚饭、附近美食或某个品类后，如果我没有马上决定去哪吃，我希望系统能帮我找回“我之前就想吃过”的选择，而不是让我重新比较所有陌生餐厅。

### 5.2 核心假设

| 编号 | 假设 | 验证方式 |
| --- | --- | --- |
| H1 | 搜索后未转化用户存在本地生活决策缺口 | 对比搜索后回流用户的后续行为流失率 |
| H2 | 高置信收藏 POI 比普通搜索后推荐更能促成决策 | BiteBack vs 普通搜索后推荐卡 |
| H3 | 收藏记忆解释能降低广告感和位置 creepy 感 | 有解释层 vs 无解释层负反馈对比 |
| H4 | 不让券参与排序能更干净地验证产品机制 | 有券/无券 POI 分层观察 |
| H5 | BiteBack 能带来增量，而不是截胡搜索页自然转化 | Eligible-level random holdout |

## 6. 收藏意图分层

收藏不等于到店意图。首发必须过滤云收藏。

| 等级 | 定义 | 权重 | 首发处理 |
| --- | --- | --- | --- |
| S | 收藏后打开过 POI/团购/路线，或搜索过同店/同品类 | 1.0 | 可召回 |
| A | 收藏 + 复看/完播/分享/评论想吃，且 POI 高置信 | 0.8 | 可召回 |
| B | 仅收藏，但视频强 POI、店铺可行动、与当前 query 强匹配 | 0.5 | 可召回 |
| C | 只收藏泛美食/合集/云吃饭内容，无 POI 或弱匹配 | 0 | 不召回 |

记忆强度计算：

```text
memory_strength =
收藏 0.30
+ 复看/完播 0.20
+ POI/团购页历史点击 0.25
+ 同品类/同店搜索 0.15
+ 近期互动衰减修正 0.10
- 云收藏惩罚
```

## 7. 六大准入 Gate

BiteBack 不是运营手动插卡，必须经过六个 Gate。任何一个 Gate 不通过，都不展示。

### 7.1 Eligibility Gate

判断用户是否属于 P0 Eligible UV。

必过条件：

- 搜索后 15 分钟内回 Feed。
- query 是美食决策意图。
- 搜索链路未完成 POI/团购/路线行为。
- 存在 S/A/B 收藏 POI。
- query 与 POI 至少一维强匹配。

### 7.2 Attribution Gate

判断 BiteBack 是否有可能产生增量，而不是抢已有转化。

必过条件：

- 搜索结果页没有完成 POI 打开、团购打开、路线点击。
- 实验组和对照组券供给一致。
- 对照组只继续普通 Feed，不展示 BiteBack。
- 指标单独观察搜索链路是否被 cannibalize。

### 7.3 Feed Guardrail Gate

判断是否适合在当前 Feed 会话插入。

不展示条件：

- 用户刚打开 App，未消费至少 2 条普通视频。
- 用户连续快速划走。
- 用户正在强沉浸观看长视频。
- 当前 session 已出现广告、本地生活组件或搜索承接组件。
- 用户近期对商业/位置相关内容有负反馈。

### 7.4 Quality Gate

保证供给可信。

必过条件：

- `poi_confidence >= 0.85`。
- 店铺营业或即将营业。
- 店铺可达。
- 无停业、退款、投诉、差评异常。
- 视频 POI 与真实门店一致。
- AIGC 文案不得编造用户原话、价格、距离、优惠。

### 7.5 Frequency Gate

控制打扰。

规则：

- 单用户每天最多 1 次。
- 同 query 只出现 1 次。
- 同 POI 7 天内最多 1 次。
- 用户负反馈后立即冷却。
- 用户关闭收藏唤醒后不再展示。

### 7.6 Business Gate

避免商业化污染产品判断。

规则：

- 券不参与首发排序。
- 券只能弱露出为“有可用团购”。
- 支付/核销 lift 不能主要由补贴解释。
- 商家出价不得覆盖用户历史兴趣和 query 匹配。

## 8. AI 与推荐策略

### 8.1 AI 不是只写文案

BiteBack 的 AI 链路：

```text
内容理解
  -> POI 归因
  -> Food Memory Asset
  -> Query-Memory 匹配
  -> 候选召回
  -> 排序
  -> 解释生成
  -> 反馈学习
```

### 8.2 Food Memory Asset

```json
{
  "memory_id": "m_001",
  "video_id": "v_001",
  "poi_id": "poi_001",
  "shop_name": "老巷牛肉面",
  "dish_tags": ["牛肉面", "红油抄手"],
  "category": "面馆",
  "business_area": "南科大",
  "price_band": "0-50",
  "memory_level": "A",
  "memory_strength": 0.82,
  "poi_confidence": 0.91,
  "last_interaction_days": 43,
  "is_open": true,
  "distance_m": 900,
  "deal_available": true
}
```

### 8.3 Query-Memory 匹配

至少满足一维强匹配：

| 匹配维度 | 示例 | 说明 |
| --- | --- | --- |
| 商圈匹配 | query=南科大附近晚饭，POI=南科大 1km 内 | 强 |
| 品类匹配 | query=面馆/晚饭，POI=牛肉面 | 强 |
| 店名匹配 | query=老巷牛肉面，POI=同店 | 最强 |
| 菜品匹配 | query=牛肉面，POI 菜品标签=牛肉面 | 中强 |

### 8.4 触发策略

```python
show_card = (
    eligibility_gate_pass
    and attribution_gate_pass
    and feed_guardrail_gate_pass
    and quality_gate_pass
    and frequency_gate_pass
    and business_gate_pass
)
```

插入策略：

- 用户回 Feed 后至少消费 2 条普通视频。
- 第 3 条后进入候选插入队列。
- 如用户强沉浸，延后。
- 如用户快速划走或当前 session 商业组件过多，不插。

### 8.5 排序策略

```python
rank_score = (
    memory_strength * 0.30 +
    query_match_score * 0.25 +
    poi_confidence * 0.15 +
    distance_fit * 0.10 +
    shop_quality * 0.10 +
    actionability * 0.10
)
```

注意：券不进入首发排序权重。

### 8.6 解释生成

文案必须短、准、可解释。

可用模板：

- “你刚搜过晚饭，这家是你收藏过的南科大附近面馆。”
- “你收藏过这家，和刚才搜的牛肉面匹配。”
- “上次收藏的这家现在营业，离你约 900 米。”

禁止：

- “你刚经过这里。”
- “我们发现你在南山。”
- 编造“你曾说过想吃”。
- 用“限时快抢”作为主文案。

## 9. 产品形态与状态机

### 9.1 卡片首屏结构

```text
┌──────────────────────────────┐
│ 你刚搜过晚饭 · 这家你收藏过  │
├──────────────────────────────┤
│ 原收藏视频封面 / 3 秒静音预览 │
│ 老巷牛肉面 · 南科大附近       │
├──────────────────────────────┤
│ 收藏第 43 天 · 约 900m        │
│ 人均 ¥38 · 当前营业中         │
│ 有可用团购                    │
├──────────────────────────────┤
│ [再看视频] [看店铺] [加入今天]│
│ 为什么推荐给我？              │
└──────────────────────────────┘
```

### 9.2 解释层

展示：

- 你刚搜索了“南科大附近晚饭”。
- 这家店来自你 43 天前收藏的视频。
- 它与 query 在商圈和品类上匹配。
- POI 置信度高，当前营业。
- 可关闭收藏唤醒。

### 9.3 负反馈

| 反馈 | 系统行为 |
| --- | --- |
| 太远了 | 收紧距离阈值，当前 POI 降权 |
| 不想吃这类 | 当前品类 7 天降权 |
| 已经吃过 | 标记已履约，进入评价/打卡链路 |
| 不是这家店 | 记录 POI 错配，当前 POI 下线 |
| 别再提醒收藏 | 关闭 BiteBack 收藏唤醒 |

### 9.4 状态机

```text
SEARCH
  -> RETURN_FEED
  -> GATE_CHECK
  -> GATE_BLOCK / CARD_QUEUED
  -> CARD_EXPOSED
  -> CARD_EXPLAINED / CARD_DISMISSED
  -> DECISION_ACTION
  -> NEGATIVE_FEEDBACK / COOLDOWN
```

## 10. 指标体系

### 10.1 北极星

> 收藏 POI 唤醒后的 7 日增量核销用户数 / Eligible UV

### 10.2 指标口径

| 指标 | 分子 | 分母 | 时间窗 | 用途 |
| --- | --- | --- | --- | --- |
| Eligible UV | 满足 P0 与六大 Gate 前置条件用户 | 搜索后回流用户 | session | 实验分母 |
| Exposure Rate | BiteBack 曝光 UV | Eligible UV | session | 曝光控制 |
| Valid Wake Rate | 回看视频/解释层/加入今天任一行为 UV | BiteBack 曝光 UV | 24h | 有效唤醒 |
| Decision Rate | POI/团购页/路线点击 UV | BiteBack 曝光 UV | 24h | 进入决策 |
| Pay Lift | 实验组支付率 - 对照组支付率 | Eligible UV | 7d | 交易增量 |
| Redeem Lift | 实验组核销率 - 对照组核销率 | Eligible UV | 7d | 履约增量 |
| Negative Rate | 负反馈 UV | BiteBack 曝光 UV | session | 打扰判断 |
| Feed Time Delta | 实验组 Feed 时长 - 对照组 Feed 时长 | Eligible UV | session/1d | Feed 护栏 |
| Privacy Complaint Rate | 隐私相关反馈 UV | BiteBack 曝光 UV | 7d | 安全护栏 |

### 10.3 不接受的成功

- CTR 高但 POI、支付、核销无增量。
- 支付 lift 来自强券补贴。
- BiteBack 抢走搜索页本来会发生的自然转化。
- 本地生活指标提升但 Feed 停留下降。
- POI 错配、隐私反馈或低质商家投诉上升。

## 11. 实验设计

### 11.1 实验对象

仅 Eligible UV。

### 11.2 分组

| 组别 | 体验 |
| --- | --- |
| Treatment | 搜索后回流，Gate 通过后展示 BiteBack |
| Control | 搜索后回流，不展示 BiteBack，继续普通 Feed |
| Optional Benchmark | 展示普通搜索后推荐卡，用于比较“收藏记忆”是否有额外价值 |

### 11.3 归因要求

- Eligible-level random holdout。
- 实验组和对照组券供给一致。
- 排除搜索链路已完成转化用户。
- 单独监控搜索页转化是否被 cannibalize。
- 按 query 意图、收藏等级、匹配类型分层看 lift。

### 11.4 Feed 护栏

任一显著劣化即 no-go：

- 推荐流停留。
- 会话深度。
- 快速划走。
- 负反馈。
- 隐私反馈。
- 广告疲劳。

## 12. 黑客松 Demo 范围

Demo 不证明真实核销，只证明“刷到那一刻成立”和“产品/数据闭环成立”。

### 12.1 必须展示

- 仿抖音 Feed。
- 搜索“南科大附近晚饭”。
- 搜索后不转化，回到 Feed。
- 六大 Gate 判断。
- 第 3 条普通视频后出现 BiteBack。
- 解释层。
- 加入今天 / 看店铺。
- 指标面板更新。
- Control 组不展示 BiteBack。
- 不触发 case：非美食 query、C 级收藏、低置信 POI、已转化、强沉浸。
- 负反馈后降权和冷却。

### 12.2 不做

- 真实支付。
- 真实地图。
- 真实定位。
- 真实抖音登录。
- 真实后端推荐模型。
- 饭点/附近/外出多场景。

## 13. 最终 Pitch

对用户：

> “你刚搜的晚饭，其实收藏夹里早有一个答案。”

对评委：

> “BiteBack 不是美食推荐，而是搜索后未完成决策时，用 AI 把用户过去在抖音沉淀的收藏兴趣重新带回 Feed，完成一次可解释、可行动、可验证增量的本地生活承接。”

对业务：

> “BiteBack 的目标是提升搜索后未转化用户的 POI 决策和 7 日增量核销，同时把 Feed 护栏放在本地生活转化之前。它是抖音从内容种草走向本地履约的前置决策实验。”
