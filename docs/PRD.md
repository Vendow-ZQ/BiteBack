## 0. 本版方向纠偏

上一版把 BiteBack 过度收敛成“搜索后未转化承接”，这个方向不够贴近功能原意。搜索可以作为一个意图信号，但不应该成为 BiteBack 的主舞台。

BiteBack 的原始命题是：

> 用户曾经在抖音收藏过某个美食内容、门店或菜品，但当时因为距离、时间、刚吃过饭、同行选择、预算或场景不合适，没有立刻去。等他之后真的处在可行动的时间和地点时，抖音把这些“曾经想吃”的收藏自然拎回 Feed，形成一张无感、好看、可行动的本地生活卡片。

因此，本版 P0 不再是“搜索后回 Feed 才触发”，而是：

> 用户处在饭点或出行/停留场景，附近存在多个高置信、可到店、来自其历史收藏的美食 POI；Feed 当前可插入时，BiteBack 在普通视频之间自然出现一张“收藏觅食卡”，帮助用户从收藏夹里快速恢复决策。

本版项目重点：

- **卡片版面设计**：卡片怎么自然出现，第一眼是否像抖音内容，而不是广告。
- **收藏资产重组**：不是推一个旧视频，而是把收藏视频背后的门店、菜品、距离、营业状态和用户记忆整理成决策卡。
- **多候选决策**：用户附近可能有多家收藏店，卡片应支持 2-3 家对比，而不是单点推荐。
- **无感触发**：用户不需要先搜索；搜索只是辅助信号。
- **可解释但不打扰**：说明“来自你的收藏”和“现在适合去”，避免 creepy 的位置表达。

## 1. 产品定义

### 1.1 一句话

BiteBack 是抖音 Feed 里的 AI 收藏美食自然唤醒卡：当用户在合适的时间和地点刷 Feed 时，把他过去收藏过、现在附近可行动的美食门店重新组织成一张好看的决策卡。

### 1.2 用户感知

用户看到的不是“系统推荐了一个广告”，而是：

> “我之前收藏过的想吃清单，刚好现在有几家就在附近。”

### 1.3 业务定位

BiteBack 是抖音本地生活从“内容种草”到“到店履约”的中间层。它不替代搜索、不替代团购频道、不替代地图，而是在 Feed 里把用户过去沉淀的收藏资产转化为当前可行动的到店意愿。

### 1.4 产品哲学

收藏不该吃灰。美食收藏夹长期沉睡，对三方都是损失：

- 对用户：明明曾经心动过，却在真正要吃饭时想不起来，只能重新搜索陌生选择。
- 对商家：已经被内容种草命中的高意愿用户没有被带到店。
- 对平台：抖音完成了种草，却没有把这份内容资产自然承接到本地生活增长。

BiteBack 的价值是四两拨千斤：不重新造一个点评列表，不强塞一个广告位，而是在用户刚好有时间、刚好在附近、刚好需要吃饭时，把“你曾经想吃的”重新递到 Feed 里。

产品文案要保留抖音感，直给、有梗、轻松，但不油腻。例如：

> 之前眼馋的，现在能吃上！

这句话背后的用户感受是：“这就是我之前收藏过的啊，现在它真的就在附近。”

### 1.5 不是什么

- 不是随机附近推荐。
- 不是进店后电子围栏推券。
- 不是单纯复播收藏视频。
- 不是美团/点评式列表页。
- 不是强商业广告位。
- 不是必须搜索后才触发的承接卡。
- 不是美团/点评式评分排行。
- 不是低价券抢购页。

### 1.6 是什么

- 是 Feed 中自然出现的收藏夹再唤醒组件。
- 是用户曾经 mark 的美食兴趣，在时间/地点合适时重新出现。
- 是门店、菜品、收藏记忆和行动信息的 AI 重组。
- 是能展示多个“我想吃过”的附近选择的轻决策卡片。

## 2. 用户需求

### 2.1 原始用户故事

用户在晚上刷到美食视频，觉得想吃，于是收藏。但当时可能在家、离门店远、刚吃完、没朋友同行，或只是在云种草。几天或几周后，用户真的出门了，可能在商圈、地铁、打车、咖啡厅、校园附近，突然需要决定吃什么。此时他不一定记得自己收藏过哪些店，也不一定愿意打开收藏夹慢慢翻。

BiteBack 要做的是在这个时刻把收藏夹中“现在可去”的美食拎出来，而不是让用户重新去美团/大众点评筛选陌生店。

### 2.2 JTBD

当我在饭点或出门途中刷抖音时，如果我附近正好有我以前收藏过、一直想吃但没去成的店，我希望抖音自然提醒我，并帮我快速判断今天去哪家。

### 2.3 用户痛点

| 痛点 | 说明 | BiteBack 解法 |
| --- | --- | --- |
| 收藏后遗忘 | 收藏夹是沉淀池，不是行动入口 | Feed 自动唤醒高意愿收藏 |
| 当时不方便去 | 收藏发生在非行动场景 | 结合当前时间/地点重新判断 |
| 不想重新搜索 | 到饭点才想吃什么，决策成本高 | 给出 2-3 个已收藏选项 |
| 怕广告感 | 附近推荐容易像商业投放 | 以“你收藏过”为主解释 |
| 单店推荐太武断 | 用户真实决策常在几家之间比较 | 卡片支持多店横向对比 |

## 3. P0 场景

### 3.1 主场景：饭点附近收藏唤醒

用户在 11:00-13:30 或 17:00-21:00 之间刷 Feed，当前位置附近 3km 内存在多个历史收藏美食 POI，且这些 POI 当前营业、质量可信、距离可达。系统在 Feed 中插入 BiteBack 卡。

### 3.2 次场景：外出停留收藏唤醒

用户出现在商圈、学校、地铁站、景区、办公区等可消费场景，并出现停留或低速移动。附近存在历史收藏门店，且当前时间接近餐饮决策窗口。BiteBack 可以触发。

### 3.3 辅助场景：搜索/评论/收藏加权

如果用户刚搜索过“附近晚饭”“火锅”“南科大美食”等，或最近复看/评论过美食内容，这些行为只作为排序加权，不再是 P0 必要条件。

### 3.4 暂不做

- 已进入店内后的推券提醒：交给电子围栏、蓝牙、WiFi 到店营销能力。
- 通用附近陌生店推荐：与收藏唤醒主命题不一致。
- 纯低价券拉转化：会污染功能判断。
- 独立地图/点评列表页：不符合 Feed 轻决策体验。

## 4. 与进店推券能力的边界

| 能力 | 触发位置 | 用户状态 | 目标 | 用户感知 |
| --- | --- | --- | --- | --- |
| 电子围栏 / WiFi / 蓝牙推券 | 用户进入或接近门店 | 已到店或强接近 | 临门一脚转化 | “这家店有券” |
| BiteBack | 用户在附近可消费区域刷 Feed | 还没决定去哪吃 | 把收藏意愿恢复为决策 | “我收藏过的店现在适合去” |

BiteBack 发生在到店前，是前置决策层。它不依赖用户已经进入店铺，也不以券为主信息。

## 5. 核心体验原则

### 5.1 自然

卡片必须像 Feed 中的一条高级内容单元，而不是弹窗、浮层或广告。它可以占据一屏，但入场方式要克制：出现在普通视频之间，有轻微“收藏夹记忆被唤醒”的视觉语言。

### 5.2 好看

卡片是项目展示重点。它要有明确版面层次：标题、主视觉、多店候选、推荐理由、行动按钮、反馈入口。不要堆满参数，不要像调试面板。

### 5.3 可行动

用户看到后 3 秒内要知道：

- 这是我收藏过的。
- 附近有几家现在能去。
- 每家吃什么、远不远、开不开、贵不贵。
- 下一步能做什么。

### 5.4 可解释

解释要建立信任，不展示“我们监控你的位置”。推荐理由应表达为：

- 来自你的收藏。
- 现在是饭点。
- 附近可达。
- 当前营业。

### 5.5 不商业污染

团购券可弱露出，但不能成为主标题、主排序和主按钮。首发要验证“收藏意愿 + 当前可行动”是否带来增量。

## 6. 收藏资产模型

### 6.1 收藏不等于旧视频

用户收藏的是内容，但产品要解析出背后的 Food Memory：

```text
收藏视频
  -> 识别菜品、门店、POI、商圈、价格、达人描述
  -> 判断用户收藏强度
  -> 形成可被重新召回的 Food Memory Asset
```

### 6.2 Food Memory Asset

```json
{
  "memory_id": "m_001",
  "source_video_id": "v_001",
  "source_cover": "/assets/covers/20260528-162431.jpg",
  "poi_id": "poi_001",
  "shop_name": "京鼎香·铜锅涮羊肉",
  "signature_dishes": ["铜锅涮羊肉", "麻酱烧饼"],
  "category": "火锅",
  "business_area": "南科大 / 宝能城",
  "saved_at_days_ago": 18,
  "memory_strength": 0.86,
  "memory_level": "A",
  "poi_confidence": 0.93,
  "shop_quality": 0.89,
  "price_per_person": 85,
  "distance_m": 395,
  "walk_minutes": 6,
  "is_open": true,
  "deal_available": true,
  "why_saved": "你上次收藏了这条探店视频"
}
```

### 6.3 收藏强度

| 信号 | 权重 | 说明 |
| --- | --- | --- |
| 收藏 | 0.30 | 用户显式 mark |
| 完播/复看 | 0.20 | 内容吸引力 |
| 打开 POI/团购页 | 0.20 | 到店意愿更强 |
| 分享/评论“想吃” | 0.15 | 社交表达 |
| 最近同品类行为 | 0.10 | 当前口味趋势 |
| 时间衰减修正 | 0.05 | 近期更强，长期保留弱记忆 |

### 6.4 收藏分层

| 等级 | 定义 | P0 处理 |
| --- | --- | --- |
| S | 收藏后曾打开 POI、团购、路线或多次复看 | 强召回 |
| A | 收藏 + 高完播/复看/分享，POI 置信高 | 可召回 |
| B | 仅收藏，但门店和菜品非常明确 | 附近强匹配时召回 |
| C | 泛美食合集、无门店、弱 POI、云吃饭 | 不召回 |

## 7. 触发与推荐逻辑

### 7.1 触发不是搜索，而是 Context x Memory x Feed

```text
是否展示 BiteBack =
用户有高意愿收藏
AND 当前时间适合餐饮决策
AND 当前地点附近有可去收藏店
AND Feed 当前可插入
AND 门店质量与隐私频控通过
```

### 7.2 六大 Gate

| Gate | 判断问题 | 通过标准 |
| --- | --- | --- |
| Memory Gate | 有没有值得唤醒的收藏 | 存在 S/A/B 收藏美食 POI |
| Context Gate | 现在是不是可行动时刻 | 饭点/外出停留/低速移动/商圈附近 |
| Proximity Gate | 收藏店是否在附近可达 | 3km 内或 20 分钟可达，至少 2 家候选更佳 |
| Feed Gate | 当前 Feed 是否适合插卡 | 已消费 2 条以上普通内容，非快速划走 |
| Quality Gate | 门店与 POI 是否可信 | POI 置信、营业、质量、风险通过 |
| Frequency & Privacy Gate | 是否打扰/敏感 | 当日不超过 1 次，可解释且可关闭 |

### 7.3 候选召回

候选来自用户过去 180 天收藏的美食内容。召回条件：

- 能绑定明确 POI。
- POI 当前可营业或即将营业。
- 用户当前位置可达。
- 最近没有负反馈。
- 非广告强推商家。

### 7.4 排序

```text
rank_score =
memory_strength * 0.28
+ context_fit * 0.20
+ proximity_fit * 0.18
+ poi_confidence * 0.12
+ shop_quality * 0.10
+ dish_diversity_fit * 0.07
+ actionability * 0.05
```

说明：

- `context_fit`：饭点、天气、同行、停留、搜索等上下文匹配。
- `proximity_fit`：距离、步行/打车时间、是否顺路。
- `dish_diversity_fit`：避免三家都是同一种菜，提升选择感。
- `deal_available` 不进入主排序，只做弱露出。

### 7.5 多店策略

P0 不只推一家。默认卡片展示 3 家候选：

- 1 家主推：综合分最高。
- 2 家备选：距离近 / 品类不同 / 价格不同。

用户可以横滑、换一批、展开对比。这样更贴近真实“今天吃什么”的决策，而不是系统替用户拍板。

## 8. 卡片版面设计

### 8.1 卡片名称

Feed 中展示名：**之前眼馋的，现在能吃上！**

内部组件名：`BiteBackCollectionCard`

### 8.2 官方示例视觉拆解

用户给的官方示例不是传统卡片，而是一个 **Feed 原生 AI 卡组**。它的架构可以拆成 7 层：

| 层级 | 示例里的表现 | 对 BiteBack 的启发 |
| --- | --- | --- |
| Feed 外壳 | 顶部频道、搜索、底部 Tab 都保留 | 让卡组仍然像抖音 Feed，而不是跳到独立工具页 |
| 沉浸背景 | 莫干山茶园大图 + 深色渐变 | BiteBack 可用食物/街区/门店氛围图做底图 |
| 大标题 | “莫干山清明越野跑攻略” | 标题必须一句话说清场景，不写算法术语 |
| AI 标识 | 小型“AI生成”标签 | 可用“AI整理”弱提示，不能抢主视觉 |
| 主信息卡 | 半透明玻璃卡，承载目的地/天数/天气/路线 | BiteBack 主卡承载收藏店数、距离、饭点、主推店 |
| 内容证据 | 博主攻略、视频缩略图、引用评论 | BiteBack 展示收藏来源、达人原视频、用户当时收藏理由 |
| 底部操作 | 不感兴趣 / 马上开吃 / 上滑继续看视频 | BiteBack 保留 Feed 控制感，同时承接即时行动 |

三张示例图分别代表三种页型：

| 页型 | 信息结构 | 设计价值 |
| --- | --- | --- |
| 总览页 | 关键事实 + AI 推荐语 + 社会证据 | 快速判断“这件事值不值得看” |
| 路线页 | 专业参数 + 可视化曲线 + 备选路线 | 把复杂决策可视化，降低理解成本 |
| 活动页 | 多个活动卡 + 图片 + 推荐评论 | 从单点推荐扩展成可行动清单 |

这说明 BiteBack 不应该只是一张静态单店卡，而应该是一个可左右滑动的 **收藏美食决策卡组**。同一张 Feed 卡内可以有多个版，每个版解决一个决策问题。

### 8.3 BiteBack 卡组结构

Feed 中展示名：**之前眼馋的，现在能吃上！**

推荐形态：全屏 Feed 卡组，左右滑动切换 3-4 个信息版；上下滑动仍然继续 Feed。

```text
┌────────────────────────────────┐
│ 抖音 Feed 顶部频道 / 搜索        │
├────────────────────────────────┤
│ 背景：附近街区/菜品氛围图 + 渐变  │
│ 大标题：之前眼馋的，现在能吃上！  │
│ AI整理                            │
├────────────────────────────────┤
│ Page 1 之前眼馋的，现在能吃上！   │
│ Page 2 今晚行动：马上开吃/约人/提醒│
│ Page 3 怎么吃最顺：路线/时间       │
│ Page 4 为什么是它们：收藏证据      │
├────────────────────────────────┤
│ 分页进度条                        │
│ [不感兴趣] [马上开吃]              │
│ 上滑继续看视频                    │
└────────────────────────────────┘
```

### 8.4 卡组页型设计

| 页 | 页面名 | 解决的问题 | 核心内容 |
| --- | --- | --- | --- |
| P1 | 之前眼馋的，现在能吃上！ | 我现在可以吃什么 | 1 主推 + 2 备选收藏店，距离、菜品、人均、营业 |
| P2 | 今晚行动 | 我下一步能做什么 | 马上开吃、发给朋友、提醒稍后、看店铺、弱团购/订座 |
| P3 | 怎么吃最顺 | 去哪家最方便 | 步行/打车时间、路线小图、排队/营业到几点、顺路程度 |
| P4 | 为什么是它们 | 为什么不是广告 | 收藏来源、原视频缩略图、达人/评论证据、收藏时间 |

P0 Demo 至少做 P1-P3，P4 可作为详情扩展。

### 8.5 P1 之前眼馋的，现在能吃上！版面

P1 的视觉采用 **沉浸背景大图 + 三图拼贴卡**：

- 全屏背景使用主推菜品或门店氛围图，叠加深色渐变。
- 中部内容卡采用三图拼贴：主推店大图 + 两个备选店小图。
- 没有真实素材时，用 `placeholderColor` 色块占位，不临时塞网图。
- 先让用户馋，再让用户看到“很近、能去、收藏过”。

```text
┌────────────────────────────────┐
│ 背景：主推菜品/门店氛围图 + 渐变 │
│ 之前眼馋的，现在能吃上！ AI整理 │
│ 从收藏夹里捞出 3 家，现在能去   │
├────────────────────────────────┤
│ [主推大图：铜锅涮羊肉]           │
│ 京鼎香·铜锅涮羊肉                │
│ 395m · 步行 6 分钟 · 营业到22:00 │
├────────────────────────────────┤
│ [桃喜甜品] 1.2km  [奶龙炒饭]800m │
├────────────────────────────────┤
```

### 8.9 版面模块职责

| 模块 | 用户看到什么 | 设计目标 |
| --- | --- | --- |
| 沉浸背景 | 附近街区/主推菜品氛围图 | 保留 Feed 内容感 |
| 大标题 | “之前眼馋的，现在能吃上！” | 第一眼命中食欲和机会感 |
| 情境副标题 | “你收藏过的 3 家，现在都在附近” | 解释为什么现在出现 |
| 关键事实条 | 饭点、商圈、筛选出的收藏数 | 建立当下行动场景 |
| 主推门店卡 | 店名、菜品、距离、营业、人均 | 3 秒内形成行动判断 |
| 备选店区 | 另外 2 家收藏店 mini row | 支持真实比较 |
| 路线/顺路页 | 步行时间、排队风险、营业窗口 | 降低到店成本 |
| 收藏证据页 | 原视频、达人、收藏时间 | 降低广告感 |
| CTA 区 | 不感兴趣 / 马上开吃 | 行动但不强迫 |

### 8.10 视觉风格

- 背景：深色半透明卡面，贴合抖音 Feed 夜间视觉。
- 图片：真实菜品图优先，门店图辅助。
- 圆角：8px 以下，保持抖音工具感和内容感。
- 字体：主标题 18-20px，门店名 17-18px，标签 12-13px。
- 动效：卡片入场轻微上滑，不弹窗，不遮挡当前视频互动按钮。
- 横滑：底部分页条类似官方示例，清楚告诉用户还有其他版。
- 页面密度：每页只解决一个问题，避免在首屏堆满所有信息。
- 商业信息：团购只能以小标签出现，例如“有团购”，不能红色大促。

### 8.11 单店与多店形态

| 形态 | 使用条件 | UI |
| --- | --- | --- |
| 多页多店卡组 | 附近有 2-3 家高质量收藏 | 默认 P0 |
| 单店卡 | 只有 1 家强匹配收藏 | 主视觉 + 大信息区 |
| 轻提醒条 | Feed 风险较高但可提醒 | 视频底部轻条，不占整屏 |

黑客松 Demo 重点展示多页多店卡组。

### 9.1.1 可衍生链接与板块

BiteBack 的卡片不只承担“看店铺”，它可以衍生出多个轻链路，但必须服务收藏唤醒主线。

| 链接/板块 | 入口位置 | 增量价值 | 风险控制 |
| --- | --- | --- | --- |
| 原收藏视频 | 收藏证据页缩略图 | 把内容种草资产重新消费 | 不作为主 CTA |
| POI Sheet | 店名 / 行动 Sheet 内入口 | 进入本地生活决策 | 不直接跳支付 |
| 模拟路线 | 怎么吃最顺页 | 提升到店意图 | 不做真实地图依赖 |
| 马上开吃 | 底部主 CTA | 把兴趣直接承接到行动 Sheet | 不直接支付，不强推券 |
| 加入今晚 | 今晚行动页次级入口 | 把兴趣变轻计划 | 不强推下单 |
| 发给朋友 | 今晚行动页 | 解决同行决策 | 用户主动触发 |
| 稍后提醒 | 今晚行动页 | 覆盖打车/地铁途中 | 频控，默认关闭 |
| 换一批收藏 | P1 收藏唤醒页 | 提升选择感 | 只从收藏池召回 |
| 有可用团购 | 详情 Sheet 小标签 | 交易承接 | 不参与排序、不抢主视觉 |

### 9.2 解释层

解释层不是算法说明书，而是信任说明：

```text
为什么出现？

1. 这几家来自你的美食收藏。
2. 现在接近晚饭时间。
3. 它们在你附近可达，且当前营业。
4. 你可以关闭“收藏唤醒”。
```

避免：

- “我们检测到你刚到南山区。”
- “你经过了这家店。”
- “大数据发现你饿了。”

### 9.3 门店详情 Sheet

点击门店或“看路线”后，不跳出 Feed，只打开底部 Sheet：

- 店名、人均、距离、营业状态、评论摘要。
- 你收藏的视频缩略图。
- 推荐菜。
- 可用团购弱露出。
- CTA：马上开吃、模拟路线、看店铺、加入今晚。

### 9.4 马上开吃 Sheet

“马上开吃”不是直接支付，而是一个行动聚合 Sheet。它负责把用户从“想吃”推进到“现在可以行动”：

已定方案：**A 路线可达优先型**。Sheet 首屏先证明“它真的离你很近、现在去不麻烦”，再补充抖音内容证据和弱商业承接。不要一打开就放大券、评分或长评论。

信息优先级：

1. 地点和可达性：让用户感觉它真的唾手可得。
2. 营业和时间确定性：现在去不会扑空。
3. 评论区和达人原话：来自抖音内容生态的真实种草证据。
4. 活动和券：弱露出，不抢主按钮。
5. 底部行动：看路线、打开店铺、发给朋友。

首屏版面固定为：

| 区块 | 内容 | 设计目标 |
| --- | --- | --- |
| 顶部店名 | 店名 + 招牌菜 + “你收藏过”轻标签 | 让用户知道这是哪家、为什么出现 |
| 距离强提示 | `离你 395m · 步行 6 分钟` | 第一眼建立唾手可得 |
| 路线口语化 | `从南科大地铁站 A 口出发，过一个路口就到` | 比地图参数更像真实建议 |
| 路线卡 | 简化地图 / 抽象路线图 / 后续可接地图 API | 给前端展示重点，也给后续真实地图留接口 |
| 时间确定性 | 营业到几点、预计到店、排队风险 | 降低“去了会不会扑空”的犹豫 |
| 内容证据 | 评论区一句 + 达人原话一句 | 维持抖音内容生态可信度 |
| 弱商业 | 有可用团购、小额代金券 | 只做助攻，不抢主线 |
| 底部动作 | 看路线 / 打开店铺 / 发给朋友 | 承接到店、POI、社交决策 |

```text
马上开吃

京鼎香·铜锅涮羊肉
你 18 天前收藏过 · 招牌：铜锅涮羊肉
离你 395m · 步行 6 分钟
从南科大地铁站 A 口出发，过一个路口就到

[简化地图 / 路线卡]

营业到 22:00
现在去预计 18:42 到店
当前排队：低

评论区都在说
“麻酱太香了，羊肉看着就很新鲜”
  来自你收藏的视频评论
“南科大附近想吃热乎的，这家真挺稳”
  来自探店达人原话

有可用团购
¥100 代金券 · ¥58

[看路线] [打开店铺] [发给朋友]
```

点击“加入今晚”后进入轻量状态：

- Sheet 内按钮变为“已加入今晚”。
- 卡片底部出现“今晚想吃 1/3”。
- 用户可以继续添加另一家。

它不是完整计划工具，只是 Demo 中体现“收藏唤醒变成行动”的中间态。

### 9.5 负反馈

| 反馈项 | 系统行为 |
| --- | --- |
| 今天不想吃这个 | 当前品类 7 天降权 |
| 太远了 | 收紧距离，当前候选降权 |
| 已经吃过了 | 标记已履约，未来减少唤醒 |
| 先别打扰我 | 当前 session 冷却，不影响未来召回 |
| 不是我想收藏的店 | 记录 POI 错配 |
| 别再提醒收藏 | 关闭 BiteBack |

反馈项排序原则：轻反馈在前，永久关闭在后。BiteBack 的负反馈不是简单关闭，而是训练收藏召回模型：口味、距离、已履约、POI 错配和打扰阈值都要被学习。

## 11. 指标体系

### 11.1 北极星

> 收藏唤醒带来的 7 日增量到店/核销用户数 / Eligible UV

黑客松 Demo 不证明真实核销，但要展示指标闭环设计。

### 11.2 漏斗指标

| 指标 | 定义 | 目的 |
| --- | --- | --- |
| Eligible UV | 有可唤醒收藏且当前场景可行动用户 | 实验分母 |
| Card Exposure Rate | BiteBack 曝光 / Eligible UV | 控制曝光 |
| Valid Wake Rate | 横滑、解释、马上开吃、看路线任一行为 / 曝光 | 是否真的唤醒 |
| Deck Depth | 平均查看页数 / 曝光 | 多页卡组是否被理解 |
| Candidate Select Rate | 切换/选择候选店 / 曝光 | 多店设计是否有效 |
| Start-Eating Rate | 马上开吃 / 曝光 | 即时行动意愿 |
| Add-to-Today Rate | 加入今晚 / 曝光 | 轻计划意愿 |
| POI Intent Rate | 看店铺/路线 / 曝光 | 到店意图 |
| Save Reactivation Rate | 打开原收藏视频或收藏证据 / 曝光 | 是否唤醒历史收藏资产 |
| Share-to-Friend Rate | 发给朋友 / 曝光 | 是否解决同行决策 |
| 7d Visit/Redeem Lift | 实验组 - 对照组 | 业务增量 |
| Negative Rate | 负反馈 / 曝光 | 打扰程度 |
| Feed Time Delta | 实验组 Feed 时长变化 | Feed 护栏 |
| Privacy Complaint Rate | 隐私相关反馈 / 曝光 | 安全底线 |

### 11.3 不接受的成功

- 卡片 CTR 高，但马上开吃、看路线、POI 意图低。
- 只靠券提升支付。
- Feed 时长、完播或负反馈明显劣化。
- 用户觉得“你怎么知道我在这儿”。
- 多店卡变成美团列表，失去 Feed 内容感。

### 11.4 增量来源

BiteBack 的增量不来自“多塞一个入口”，而来自把抖音已有收藏资产在正确时刻重新激活。

| 增量来源 | 机制 | 对应指标 |
| --- | --- | --- |
| 收藏资产再激活 | 用户不用翻收藏夹，直接在 Feed 看见曾经想吃的店 | Save Reactivation Rate |
| 决策成本下降 | 3 家候选 + 路线/营业/人均，减少重新搜索和外跳 | Candidate Select / Add-to-Today |
| 本地生活承接 | 详情 Sheet、路线、POI 意图承接到店链路 | POI Intent / Visit Lift |
| Feed 留存不受损 | 卡组像内容，保留上滑继续看视频 | Feed Time Delta |
| 偏好学习 | 负反馈和候选切换反哺收藏口味、距离阈值、品类偏好 | Negative Rate / Next-session Recall |
| 社交决策 | 发给朋友降低“同行吃什么”摩擦 | Share-to-Friend Rate |

这个功能的深度在于：它不是推荐一个附近商家，而是把“用户在抖音里曾经表达过的意愿”转成当前可行动的轻决策，并把用户对卡组的交互继续沉淀回本地生活推荐模型。

## 12. 实验设计

### 12.1 分组

| 组别 | 体验 |
| --- | --- |
| Treatment | Feed 中展示 BiteBack 多店收藏卡 |

核心比较：

- 收藏等级 S/A/B。
- 距离 0-500m、500m-1.5km、1.5-3km。
- 饭点 vs 非饭点。
- 单人出行 vs 停留场景。
- 是否有最近美食搜索。
- 1 家候选 vs 2-3 家候选。

## 14. 素材与数据契约

卡片设计是本项目重点，因此未来一定会替换真实菜品图、门店图、达人头像、原收藏视频、路线图和真实门店信息。为了避免后续返工，P0 从文档和工程上就必须遵守素材契约。

### 14.1 组件不硬编码

React 组件不得直接写：

- 具体图片文件名。
- 具体视频文件名。
- 真实门店文案。
- 地图 API key。
- 外部地图 URL。
- 固定经纬度。

组件只接收 mock 数据中的字段，例如：

```ts
{
  shopName: '京鼎香·铜锅涮羊肉',
  heroAssetId: 'food-hotpot-main',
  shopAssetId: 'shop-hotpot-front',
  mapAssetId: 'map-campus-route',
  sourceVideoAssetId: 'source-video-hotpot'
}
```

再由资源 manifest 解析成路径或占位色块。

### 14.2 运行时素材目录

| 目录 | 内容 | 使用位置 |
| --- | --- | --- |
| `public/assets/food/` | 菜品、饮品、套餐图片 | P1 收藏唤醒页、主推店图、候选店图 |
| `public/assets/shops/` | 门店外立面、室内、Logo | 门店详情 Sheet、主推店环境图 |
| `public/assets/maps/` | 静态路线图、抽象地图底图 | 怎么吃最顺页 |
| `public/assets/avatars/` | 达人头像、评论头像 | 为什么是它们页 |
| `public/assets/videos/` | 原收藏探店视频、Feed 视频 | 收藏证据页、普通 Feed |
| `public/assets/covers/` | 当前 demo 兼容封面 | 旧素材兼容，后续逐步替换 |

### 14.3 占位策略

前期没有真实素材时，UI 使用色块占位即可：

- 食物占位：暖色或深红棕。
- 饮品/甜品占位：粉色或奶油色。
- 门店占位：深绿/灰色。
- 地图占位：深绿线条或抽象路径。
- 头像占位：圆形渐变。

占位色放在 mock asset manifest 的 `placeholderColor` 字段中，不写在组件里。

### 14.4 地图 API 预留

P0 不接真实地图 API，但数据结构要预留：

```ts
{
  mapAssetId: 'map-campus-route',
  routeProvider: 'mock',
  distanceText: '395m',
  durationText: '步行 6 分钟',
  routeSummary: '从南科大地铁站出发，步行可达',
  routePolylineMock: [[0.2, 0.7], [0.5, 0.5], [0.8, 0.3]]
}
```

后续接地图 API 时，替换 route provider 和 route data，不改卡片 UI 结构。

## 15. 验收标准

### 15.1 产品验收

- 评委能在 10 秒内理解：这是“我收藏过的店现在在附近”，不是广告。
- 卡片本身足够好看，是展示重点。
- 多家候选让用户有“今天吃什么”的选择感。
- 搜索不再是触发前提。
- 旧视频只是记忆来源，不是核心推荐对象。
- 券不抢主视觉。
- 解释不 creepy。

### 15.2 工程验收

- mock 数据可编辑。
- 收藏、门店、当前上下文、卡片候选分离。
- 推荐逻辑能输出多候选。
- 卡片组件能展示多店候选和详情 Sheet。
- 调试面板能展示每个 Gate 的通过/拦截原因。
- 组件不硬编码图片、视频、地图和真实门店信息。
- 素材通过 `src/mocks/assets.ts` 和 `src/config/assets.ts` 解析。
- 没有真实素材时能用色块占位。
- `npm run build` 通过。

## 16. 最终 Pitch

对用户：

> “你以前收藏过的想吃，现在刚好在附近。”
