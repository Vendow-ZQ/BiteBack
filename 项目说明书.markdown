# BiteBack 收藏美食自然唤醒卡｜项目说明书

> 赛道：AI 体验｜刷到懂你的瞬间  
> 作品形态：抖音 Feed 原生 AI 卡片功能原型  
> 提案视角：以抖音 Feed / 本地生活产品经理视角，提出一项可进入真实产品链路验证的功能更新

---

## 1. 项目概览

### 1.1 项目名称

BiteBack 收藏美食自然唤醒卡

### 1.2 一句话介绍

BiteBack 是一张自然出现在抖音 Feed 中的 AI 美食记忆卡：当用户在饭点、外出停留、附近可达的场景中刷 Feed 时，把他过去收藏过但还没去成的美食门店重新唤醒，并组织成“1 家主推 + 2 家备选”的轻决策卡片。

### 1.3 核心体验

用户不需要搜索，也不需要主动翻收藏夹。刷到 BiteBack 时，他会看到一句很直接的话：

> 之前眼馋的，现在能吃上。

卡片会告诉用户：

- 这是你以前收藏过的美食。
- 它现在就在附近，可到店。
- 当前营业，不容易扑空。
- 这里有 1 家主推和 2 家备选。
- 你可以马上开吃、打开店铺购券、发给朋友或稍后提醒。

## 2. 为什么做

### 2.1 受众是谁

首期目标用户是抖音中高频用户里，有本地生活美食收藏行为的人。

| 维度 | 说明 |
| --- | --- |
| 年龄 | 18-35 岁为主，学生、年轻白领、城市生活用户 |
| 身份 | 校园用户、通勤用户、商圈附近用户、周末出行用户 |
| 兴趣偏好 | 爱刷美食探店、饮品新品、校园食堂、商圈餐厅、达人推荐 |
| 使用场景 | 饭点刷 Feed、地铁站/商圈/校园附近停留、打车或步行途中、和朋友临时决定吃什么 |

这类用户的共同点是：他们会在抖音里收藏美食，但真正要吃饭时，常常想不起来自己收藏过什么。

### 2.2 真实观察：收藏夹里的美食经常吃灰

观察 1：

用户晚上刷到一家火锅探店，觉得很想吃，于是收藏。但当时人在宿舍或家里，刚吃完饭，不会立刻出门。几天后他刚好到商圈附近，要吃晚饭，却没有打开收藏夹，而是重新去搜索“附近美食”。

观察 2：

校园用户经常刷到“南科大附近美食”“食堂新品”“宝能城新店开业”这类内容。收藏时是兴趣表达，真正要吃饭时却是即时决策：远不远、开不开、贵不贵、朋友愿不愿意去。收藏夹没有把这些信息整理出来，所以用户还是要重新做一遍选择。

这说明收藏不是没有价值，而是缺少一个“合适时刻重新出现”的机制。

### 2.3 现有体验还缺什么

| 现有路径 | 缺口 |
| --- | --- |
| 翻收藏夹 | 内容混杂，缺少距离、营业、路线和当前可达判断 |
| 搜索附近美食 | 从陌生选项重新开始，决策成本高，也容易流向外部工具 |
| 本地生活频道 | 入口较重，用户需要主动进入 |
| 进店推券 | 发生太晚，用户已经接近门店，无法影响“去哪吃” |
| 普通附近推荐 | 容易像广告，因为缺少“这是我曾经想吃过”的理由 |

BiteBack 补的是内容种草和到店履约之间的断点。

```text
原链路：
刷到美食 -> 收藏 -> 吃灰 -> 饭点重新搜索

BiteBack 链路：
刷到美食 -> 收藏 -> 饭点/附近 -> Feed 自然唤醒 -> 路线/店铺/团购弱承接
```

## 3. 产品机会

### 3.1 功能定位

BiteBack 不是一个新的广告位，也不是搜索结果页的延伸。它是 Feed 内的本地生活轻决策卡片。

它的核心判断是：

> 用户收藏过，说明有历史意愿；现在附近可达，说明有行动条件；Feed 中自然出现，说明触达成本最低。

### 3.2 为什么它适合抖音 Feed

抖音 Feed 的优势不是让用户主动找，而是让内容在正确时刻被刷到。BiteBack 把这个优势用于本地生活：

- 抖音知道用户曾经收藏过哪些美食内容。
- 抖音可以识别内容背后的门店、菜品和 POI。
- Feed 可以在用户不主动搜索时，给出低打扰的提醒。
- 本地生活链路可以承接店铺、路线、团购和到店。

所以这不是“附近店推荐”，而是“收藏意愿的再激活”。

### 3.3 产品目标

| 目标 | 说明 |
| --- | --- |
| 用户侧 | 降低饭点决策成本，让用户想起曾经收藏过、现在能去的店 |
| 平台侧 | 把内容收藏资产转化为本地生活意图和履约增量 |
| 商家侧 | 触达曾经被种草、当前附近可行动的高意愿用户 |
| Feed 侧 | 以原生卡片形态出现，不破坏推荐流体验 |

## 4. 做了什么：三个核心功能

### 4.1 功能一：Feed 原生收藏唤醒卡

**功能名称：** 之前眼馋的，现在能吃上

**设计意图：**  
在用户饭点、外出停留、附近存在可行动收藏门店时，把历史收藏自然插回 Feed。用户刷到时不用理解复杂逻辑，只需要知道：这是我以前收藏过的，现在可以去。

**解决的问题：**

- 收藏夹不会主动提醒。
- 饭点用户容易重新搜索。
- 附近推荐缺少个人相关性，容易像广告。

**核心设计：**

- 卡片出现在普通视频之间。
- 视觉上保留 Feed 原生内容感。
- 标题用“之前眼馋的，现在能吃上”直接命中场景。
- 首屏展示主推门店、两个备选门店、距离、营业、人均和收藏来源。

### 4.2 功能二：多店轻决策卡组

**功能名称：** 1 家主推 + 2 家备选

**设计意图：**  
吃饭决策不是系统推荐一家就结束。用户通常会比较距离、品类、价格、同行偏好和顺路程度。多店卡组保留用户选择权，也让卡片更像“帮我挑出来”，而不是“替我拍板”。

**解决的问题：**

- 单店推荐太武断。
- 用户真实决策需要比较。
- 商家广告感需要被降低。

**核心设计：**

卡片分成四个页面，每页解决一个问题：

| 页面 | 解决的问题 |
| --- | --- |
| 之前眼馋的，现在能吃上 | 现在有什么能吃 |
| 今晚行动 | 下一步可以做什么 |
| 怎么吃最顺 | 去哪家最方便 |
| 为什么是它们 | 为什么不是广告 |

### 4.3 功能三：马上开吃行动闭环

**功能名称：** 马上开吃

**设计意图：**  
用户点击主 CTA 后，不直接跳支付，也不先放优惠券，而是先回答“我现在去是否方便”。这样更符合从收藏唤醒到到店行动的心理路径。

**解决的问题：**

- 用户想吃，但不确定远不远。
- 用户担心去了扑空。
- 用户需要把选择发给朋友或继续看路线。

**核心设计：**

Sheet 首屏信息顺序：

1. 店名和招牌菜。
2. 离你多远、步行多久。
3. 口语化路线。
4. 营业到几点、预计到店、排队风险。
5. 评论区和达人证据。
6. 团购弱露出。
7. 打开店铺、发给朋友；打开店铺后进入马上购券和券 / 套餐列表。

## 5. 推送判断逻辑

### 5.1 不是搜索触发，而是场景触发

BiteBack 不要求用户先搜索。搜索可以作为排序加权信号，但不是必要前提。

核心判断：

```text
是否展示 BiteBack =
用户有可唤醒收藏
AND 当前是可行动餐饮场景
AND 附近存在可达收藏门店
AND Feed 当前适合插入
AND 门店质量和 POI 置信通过
AND 频控和隐私策略通过
```

### 5.2 六大 Gate

| Gate | 判断问题 | 通过标准 |
| --- | --- | --- |
| Memory Gate | 有没有值得唤醒的收藏 | 存在高意愿、高置信、可绑定 POI 的美食收藏 |
| Context Gate | 当前是否适合餐饮决策 | 饭点、外出、停留或低速移动、处于可消费区域 |
| Proximity Gate | 收藏店是否附近可达 | 距离和通行时间在可接受范围内，且当前营业 |
| Feed Gate | 当前 Feed 是否适合插卡 | 已消费一定数量普通内容，非快速划走，非连续商业组件 |
| Quality Gate | 门店与 POI 是否可信 | POI 置信、营业状态、门店质量、风险控制通过 |
| Frequency & Privacy Gate | 是否打扰或敏感 | 当日频控、用户授权、负反馈、隐私表达均通过 |

### 5.3 排序原则

排序不以团购为主，也不以商家出价为主。首期需要验证的是“收藏意愿 + 当前可行动”是否真的带来增量。

排序因子包括：

- 收藏强度：收藏、复看、打开 POI、分享、评论等。
- 当前场景：饭点、外出、停留、最近同品类兴趣。
- 距离可达：步行时间、打车时间、是否顺路。
- POI 置信：门店识别是否准确。
- 门店质量：营业状态、评分风险、闭店风险。
- 品类多样性：避免三家候选都太像。

团购只做弱承接，不进入主排序。

## 6. 增益在哪里

### 6.1 对用户的增益

| 用户问题 | BiteBack 提供的价值 |
| --- | --- |
| 想不起来收藏过什么 | Feed 自动唤醒曾经收藏过的美食 |
| 不想重新搜索 | 直接给出已收藏、当前可去的候选 |
| 不知道远不远 | 首屏展示距离、步行时间、路线 |
| 怕广告 | 用“来自你的收藏”和原视频证据解释 |
| 同行难决策 | 支持发给朋友和多候选比较 |

### 6.2 对平台的增益

| 平台目标 | BiteBack 如何贡献 |
| --- | --- |
| 提升本地生活意图 | 在 Feed 内形成 POI、路线、店铺打开意图 |
| 提升内容资产复用 | 收藏内容不只停留在收藏夹，而是进入二次消费和决策 |
| 降低外部流失 | 用户无需跳到点评、地图重新做吃饭决策 |
| 丰富 Feed AI 卡片形态 | 提供一种可行动、可解释、可实验的 Feed 原生 AI 卡 |
| 反哺推荐模型 | 候选切换、负反馈、路线意图可以沉淀用户偏好 |

### 6.3 对商家的增益

| 商家目标 | BiteBack 如何贡献 |
| --- | --- |
| 获取高意愿用户 | 用户曾经收藏过，意愿强于陌生曝光 |
| 提升到店转化 | 卡片发生在“还没决定去哪吃”的时刻 |
| 承接团购核销 | 团购在高意愿 Sheet 中自然露出 |
| 延长达人内容价值 | 原探店内容成为用户决策证据 |
| 获得更准的流量 | 不是泛附近流量，而是收藏过、附近可达的用户 |

### 6.4 增量定义

BiteBack 的增量不应该被定义为卡片点击，而应该定义为从收藏意愿到本地生活行动的增量。

| 增量层级 | 关键指标 | 说明 |
| --- | --- | --- |
| 唤醒增量 | Valid Wake Rate | 用户是否因为卡片产生有效停留、横滑、解释或行动 |
| 决策增量 | Candidate Select Rate、Deck Depth | 多候选和多页卡组是否真的帮助选择 |
| POI 增量 | Shop Open Rate、POI Intent Rate | 用户是否进入店铺详情或 POI 链路 |
| 路线增量 | Route Intent Rate | 用户是否开始考虑真实到店路径 |
| 交易增量 | Deal View / Redeem Lift | 团购是否被高意愿用户自然承接 |
| 履约增量 | 7d Visit / Redeem Lift | 相比对照组是否带来真实到店或核销 |

## 7. Evaluation：怎么判断成功

### 7.1 北极星指标

> BiteBack 带来的 7 日增量到店 / 核销用户数 / Eligible UV

解释：

- Eligible UV 是有可唤醒收藏、当前场景可行动、且符合频控和隐私策略的用户。
- 7 日窗口覆盖当下去和稍后去两类行为。
- 使用增量而不是绝对值，避免把自然会到店的用户误算成功能贡献。

### 7.2 主指标

| 指标 | 定义 | 目的 |
| --- | --- | --- |
| Card Exposure Rate | 卡片曝光 / Eligible UV | 控制触达规模 |
| Valid Wake Rate | 横滑、解释、马上开吃、路线、店铺任一行为 / 曝光 | 判断是否真正唤醒 |
| Start Eating Rate | 点击马上开吃 / 曝光 | 判断即时行动意愿 |
| Route Intent Rate | 路线页意图 / 曝光 | 判断到店路径意图 |
| POI Intent Rate | 打开店铺或详情 / 曝光 | 判断本地生活承接 |
| Candidate Select Rate | 切换候选 / 曝光 | 判断多候选是否有价值 |
| Save Reactivation Rate | 打开原收藏视频或证据页 / 曝光 | 判断收藏资产是否被重新消费 |

### 7.3 商业指标

| 指标 | 定义 | 说明 |
| --- | --- | --- |
| Deal View Rate | 查看团购 / 曝光 | 团购弱承接能力 |
| Redeem Lift | 实验组核销 - 对照组核销 | 交易增量 |
| Merchant Visit Lift | 实验组到店 - 对照组到店 | 商家履约增量 |
| POI Follow / Save Lift | 店铺收藏、关注、二次保存增量 | 中长期商家价值 |

### 7.4 护栏指标

| 护栏 | 目标 |
| --- | --- |
| Feed Time Delta | 不显著伤害 Feed 时长 |
| Swipe-away Rate | 卡片出现后快速划走率不能异常 |
| Negative Feedback Rate | 负反馈率在可控范围 |
| Privacy Complaint Rate | 隐私相关投诉必须极低 |
| Commercial Fatigue | 用户不能感知为强广告或强推券 |
| Merchant Quality Complaint | 到店体验、闭店、POI 错配不能恶化 |

## 8. A/B 实验设计

### 8.1 实验分组

| 组别 | 体验 | 目的 |
| --- | --- | --- |
| Control | 不展示 BiteBack | 计算自然到店和自然核销基线 |
| Treatment A | 展示多店收藏卡 | 验证主方案 |
| Treatment B | 展示单店收藏卡 | 验证多候选是否必要 |
| Benchmark | 展示普通附近推荐卡 | 验证“收藏过”是否优于陌生附近推荐 |

### 8.2 实验人群

实验只在 Eligible 用户中进行：

- 有历史美食收藏。
- 收藏可绑定可信 POI。
- 当前处于饭点或可消费场景。
- 附近存在可达门店。
- 通过隐私和频控策略。

这样可以避免把不可能触发的人群放进分母，稀释实验结果。

### 8.3 分层观察

| 分层 | 目的 |
| --- | --- |
| 收藏强度 S/A/B | 判断强收藏是否更容易带来履约 |
| 距离 0-500m / 500m-1.5km / 1.5-3km | 找到可行动距离阈值 |
| 饭点 / 非饭点 | 验证 Context Gate |
| 1 家候选 / 2-3 家候选 | 验证多候选价值 |
| 有无团购 | 判断商业弱承接贡献 |
| 有无最近美食搜索 | 验证搜索作为加权信号的价值 |

### 8.4 成功标准

首期成功不要求所有指标同时最大化。建议判断顺序：

1. Feed 护栏不显著变差。
2. Privacy Complaint Rate 不上升。
3. Valid Wake Rate 显著高于普通附近推荐。
4. Route Intent / POI Intent 有显著提升。
5. 7 日到店 / 核销相对 Control 有增量。
6. 多店卡优于单店卡，或者至少不弱于单店卡且带来更高候选选择和决策深度。

## 9. 技术栈与落地方式

### 9.1 当前原型技术栈

当前原型使用：

- React 18
- TypeScript
- Vite
- 前端状态模拟 Feed、卡片交互、指标面板和实验对照

原型目标是验证产品交互、卡片结构、触发逻辑和指标闭环，不代表线上最终架构。

### 9.2 线上化依赖

如果作为抖音内部功能推进，建议拆成以下能力：

| 模块 | 说明 |
| --- | --- |
| 收藏理解服务 | 从用户收藏内容中识别菜品、门店、POI、品类、价格和达人证据 |
| 用户美食记忆资产 | 存储用户可被唤醒的美食兴趣与门店记忆 |
| Context 服务 | 判断饭点、外出、停留、商圈、天气等场景 |
| POI / 本地生活服务 | 提供营业状态、距离、团购、店铺质量、风险信息 |
| Gate & Ranking 服务 | 进行触发判断、频控、召回、排序 |
| Feed 卡片容器 | 在 Feed 中以原生卡片形态展示 |
| 埋点与实验平台 | 支持曝光、交互、路线、POI、核销和护栏指标 |

### 9.3 研发重点

| 研发点 | 要求 |
| --- | --- |
| 触发可控 | Gate 逻辑可灰度、可配置、可回滚 |
| 频控严格 | 单用户单日曝光上限明确 |
| 隐私安全 | 文案和策略避免位置监控感 |
| 卡片稳定 | 不影响 Feed 滑动、播放和性能 |
| 数据闭环 | 曝光、点击、横滑、路线、POI、团购、负反馈全链路埋点 |
| 策略可替换 | 初期规则排序，后续可接推荐模型 |

## 10. 项目思考：关键取舍与解决方法

### 10.1 它是不是搜索后的承接卡？

搜索是强意图，但如果只在搜索后出现，BiteBack 会变成搜索链路补丁。

最终取舍：

- 搜索可以作为排序加权。
- 核心触发必须是收藏记忆 + 当前可行动场景。
- 主舞台是 Feed，而不是搜索结果页。

### 10.2 它会不会像广告？

附近推荐天然容易被感知为广告。解决方法不是隐藏商业价值，而是改变推荐理由：

- 不说附近热门。
- 不说限时优惠。
- 先说来自你的收藏。
- 再说现在可达、当前营业。
- 团购只弱露出，不抢主按钮。

### 10.3 为什么不是单店推荐？

单店推荐更简单，但也更像系统拍板。多候选更接近真实吃饭决策：

- 主推降低理解成本。
- 备选保留选择权。
- 候选切换可以成为偏好学习信号。

### 10.4 为什么“马上开吃”先讲路线？

Page 1 已经解决“想不想吃”。点击 Sheet 后，用户真正关心的是：

> 我现在去麻不麻烦，会不会扑空？

因此 Sheet 首屏优先展示距离、步行时间、路线、营业和排队风险，而不是先放优惠券。

### 10.5 怎么避免位置隐私不适？

位置是触发必要条件，但不能在表达上像监控。

解决策略：

- 用“附近可达”替代“检测到你在这里”。
- 用“现在接近晚饭时间”替代“大数据猜你饿了”。
- 提供解释层和关闭入口。
- 保持严格频控。

## 11. 未来规划

### 11.1 近 3 个月

- 完善内容理解：从收藏视频中稳定解析菜品、门店、POI、品类和价格。
- 小流量实验：先在高收藏、高 POI 置信、高餐饮场景中灰度。
- 补充真实地图能力：接入距离、路线、预计到店和营业状态。
- 完善埋点：曝光、横滑、候选切换、马上开吃、路线、店铺、团购、负反馈。
- 优化解释与频控：重点控制隐私感和打扰感。

### 11.2 中长期 1 年内

- 从美食扩展到咖啡、展览、演出、运动场馆、亲子活动、周末短途游。
- 建设用户“兴趣记忆资产”能力，让收藏、复看、分享都能在未来合适场景中被重新唤醒。
- 将 BiteBack 从单张卡片扩展为一类 Feed 原生 AI 行动卡。
- 引入更精细的用户偏好学习，利用负反馈、候选切换、路线意图优化后续召回。

### 11.3 商业化与落地设想

- 首期不卖广告位，先验证收藏唤醒是否带来真实到店和 POI 增量。
- 团购、套餐、订座、排队等只在用户产生行动意图后弱承接。
- 对商家侧，可以作为“内容种草用户再触达”的高意愿转化能力。
- 对达人侧，可以延长本地生活内容价值，让收藏内容不只贡献一次播放，而是进入后续决策链路。

## 12. 最终 Pitch

### 对用户

> 之前眼馋的，现在能吃上。

### 对产品团队

> BiteBack 把收藏夹里沉睡的美食意愿，在饭点和附近场景重新带回 Feed，让用户不需要主动搜索，也能从“曾经想吃”走到“现在能去”。

### 对本地生活业务

> 它把抖音已有内容收藏资产转化为 POI 意图、路线意图、店铺打开和履约增量，同时不依赖强推券，不牺牲 Feed 体验。

### 对商家

> 它触达的不是陌生流量，而是已经被内容种草过、当前附近可达、处在消费决策窗口的高意愿用户。

### 对研发团队

> 它是一套可以被 Gate 控制、被实验验证、被推荐模型替换、被本地生活链路承接的 Feed 原生 AI 卡片能力。
